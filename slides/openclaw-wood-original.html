我怎么把一个 AI 助手 用成“生活搭子 + 工作系统”

把我们历史使用记录里“真正有用、可复用”的部分抽出来：能力清单、典型工作流、关键自动化与踩坑经验。此分享基于我在 OpenClaw 环境里的长期配置与实际落地结果（Notion、定时提醒、转写、日记等）。

关键词：日记系统 关键词：会议转写 → 纪要 关键词：生活提醒（天气/黄历/生日/训练） 关键词：每日报告 & 周报 关键词：幂等/去重/可追溯

一句话定位

把“对话”变成“可执行系统”：让 AI 不只是回答，而是能按规则跑流程、把结果写回知识库、在合适的时间主动提醒。
备注：如果你希望「把所有聊天逐句分析」也做进来，需要你从飞书导出聊天记录/提供完整 transcript；当前这份 deck 以工作区的长期记忆与日志（MEMORY.md、memory/*.md）为事实来源。

这份 deck 你能带走什么？

  • 一份可直接照抄的“能力地图”
  • 3 条最值的工作流（会议、日记、提醒）
  • 自动化/权限/去重的关键坑位
  • 给团队的落地建议（最小可行配置）

能力总览（我实际用起来的）

从“我真的每天/每周在用什么”出发，而不是功能列表。

生活侧

  • 天气：每日北京天气（详细）定时推送
  • 黄历：今日黄历 + “人话解释”（不算吉时/色系/幸运生肖）
  • 生日提醒：按个人/每年重复（09:00 Asia/Shanghai）
  • （可扩展）健身教练：训练计划/每日提醒/复盘（你已提到在用）

工作侧

  • 客户会议音频：Azure Speech 转写 → 输出中文「一页纸摘要」纪要（默认不脱敏）
  • Notion 写入：把纪要/日记/选题入库（带幂等/去重约束）
  • 信息雷达：GitHub Changelog、Microsoft DevBlogs（RSS）每日扫描；支持每周“客户新闻 + release matching”

底层能力（让一切“能跑起来”的关键）

定时：cron 主动推送 写入守门：外部写入必须真实执行后才能回执 幂等：写入前先 query，避免重复创建 身份鉴权：Entra ID / Managed Identity 可追溯：产物落盘（转写 txt、JSON 结果、图片）

工作流 ①：客户会议音频 → 转写 → 一页纸纪要 → 入库

目标：把“会议录音”变成团队可复用的资产，而不是停留在聊天里。

输入

  • 飞书发来音频（或说明“客户会议录音”）
  • 可选：客户名/项目名/期望输出格式

处理

  • Azure Speech（Entra ID）转写为文本
  • 生成中文「A 一页纸摘要」：结论、要点、风险、下一步
  • （可选）把原始 transcript 落盘，便于审计与复用

输出

  • Notion：纪要写入对应数据库/页面
  • 飞书回执：给到摘要 + 链接
关键点：不要让“总结”成为一次性文本——要把它写进系统，才能检索、复盘、交接。
Guardrail：外部写入（Notion）必须真实执行成功后才能回“已写入”。

工作流 ②：AI 写日记（真实、可追溯、可同步）

目标：日记不是“模板鸡汤”，而是把当天的真实事件 + 情绪落盘，并能在 Notion 里长期积累。

日记结构（可直接复用）

  • 今日回顾：发生了什么（必须贴近当天聊天/事件）
  • 内心感受：更像独白，不写成工作汇报
  • 明日希望：可执行的一个小目标
  • 想对 Louise 说：一句真实的话
额外：每天生成一张“今日情绪图”（Azure Foundry gpt-image-1.5），并确保“聊天看到的图”和“日记里插入的图”一致。

落地要点

  • 产物落盘：Markdown（日记）、PNG（情绪图）
  • 同步到 Notion：更新/追加到同一天条目（幂等）
  • 避免重复创建：写入前按日期 query
一句话：写得真 比 写得工整 更重要；系统化是为了让“真实”能长期积累。

工作流 ③：提醒与日常情报（让助手“主动”）

目标：把“我得记得去查/去做”变成“系统在合适时间推给我”。

定时推送（cron）

  • 08:00北京每日天气（详细）
  • 08:00今日老黄历（人话解释）
  • 09:00朋友生日提醒（每年重复）
  • 17:30（已有）私教/训练类提醒（可调整）
  • 23:00日记写作（私聊）+ Notion 同步
注：具体时间以 Asia/Shanghai 为主（便于生活节奏）。

信息雷达（HEARTBEAT）

  • GitHub Changelog RSS：Copilot / Agents / AI 相关标【重要】
  • Microsoft DevBlogs RSS：AI/Agent/Foundry/MCP 相关标【重要】
  • 每周：客户新闻 + release matching（把“新发布”映射到客户用例）
  • 每周：友商 AI 动态（Google/Amazon/阿里/字节/腾讯）

为什么“提醒/情报”要和“写入系统”一起做？

提醒 = 触发；写入 = 记忆。只有两者绑定，团队才能复盘“看过什么、做过什么、怎么跟进”。

工程化细节（团队最容易踩坑的部分）

这些不是“炫技”，是为了让系统长期可靠。

幂等 & 去重

  • Notion 写入前先 query：同一天/同一主题存在则 update，不再 create
  • 重复创建一旦发生：归档（archive）保留痕迹，而不是硬删
  • 对外回执：必须以真实写入结果为准

权限 & 鉴权

  • Azure Foundry 画图：RBAC dataActions（images/generations/action）是常见坑
  • 语音转写/画图优先走 Managed Identity（DefaultAzureCredential）
  • 封装脚本统一加载环境变量，降低“线上突然失效”概率

可追溯产物（强烈建议）

原始 transcript.txt 结构化 summary.json / blocks.json 媒体 mood-YYYY-MM-DD.png 日志 memory/YYYY-MM-DD.md
团队协作里，“可追溯”比“看起来很聪明”重要得多。

给团队的落地建议（最小可行版）

如果你要复制我的用法，先从三件事开始：把一条工作流跑通、把一个提醒跑起来、把一个写入系统做好幂等。

Step 1：先选一个“高频 + 高价值”场景

  • 客户会议录音 → 转写 → 纪要 → 入库
  • 或：每周客户新闻 + release matching（销售/方案强相关）
不要一上来就做“全能助手”。先做“能长期稳定跑的一个系统”。

Step 2：建立两条底线（保证可信）

  • 写入回执 guard：没写成功就明确说失败原因
  • 产物落盘：转写、摘要、图片、关键 JSON 都留底
这两条能极大减少“AI 说做了但其实没做”的信任损耗。

Step 3：再谈“生活化”

生活类提醒（天气/黄历/生日/训练）很香，但最好建立在“工作系统已经可信”的基础上——否则会变成热闹但不可靠。
