How I turned an AI assistant into a "life companion + work system"

This deck pulls out the genuinely useful, reusable parts from my actual usage history: the capability map, the repeatable workflows, the critical automations, and the hard-earned lessons. It is based on the long-term configuration and real results I built around OpenClaw, including Notion, scheduled reminders, transcription, and diary flows.

Keyword: Diary system Keyword: Meeting transcription -> recap Keyword: Life reminders (weather / almanac / birthdays / training) Keyword: Daily digests & weekly reports Keyword: Idempotent / deduplicated / traceable

One-line positioning

Turn "conversation" into an executable system: AI should not only answer, it should run workflows by rule, write results back into a knowledge base, and proactively remind me at the right moment.

Workflow 1: client meeting audio -> transcript -> one-page recap -> database

Goal: turn "meeting audio" into a reusable team asset instead of letting it disappear in chat.

Input

  • Audio sent from Feishu (or a note that it is a client meeting recording)
  • Optional: client name / project name / preferred output format

Processing

  • Use Azure Speech (Entra ID) to transcribe to text
  • Generate a Chinese one-page summary: conclusion, key points, risks, and next steps
  • Optional: persist the raw transcript for audit and reuse

Output

  • Notion: write the recap into the right database or page
  • Feishu receipt: return the summary and link
Key point: do not let a summary remain a disposable block of text — write it into the system so it can be searched, reviewed, and handed over.
Guardrail: external writes (Notion) should only say "written" after the real action succeeds.

Workflow 2: AI-written diary (authentic, traceable, synced)

Goal: the diary is not a template or empty comfort piece; it captures real events and emotions from the day and accumulates inside Notion over time.

Diary structure (directly reusable)

  • Today in review: what happened, grounded in the day's chats / events
  • Inner feeling: closer to a monologue than a work report
  • Hope for tomorrow: one small executable target
  • A note to Louise: one honest sentence
Extra: generate a "mood of the day" image every day with Azure Foundry gpt-image-1.5, and keep the image shown in chat consistent with the one inserted into the diary.

Implementation details

  • Persist artifacts: Markdown (diary) and PNG (mood art)
  • Sync to Notion: update or append to the same-day entry (idempotent)
  • Avoid duplicate creation: query by date before writing
One sentence:authenticity matters more than polish ; systemization exists so authenticity can accumulate over time.

Workflow 3: reminders and daily signals (make the assistant proactive)

Goal: turn "I need to remember to check / do this" into "the system pushes it to me at the right moment".

Scheduled pushes (cron)

  • 08:00Detailed daily Beijing weather
  • 08:00Today's almanac with a plain-language explanation
  • 09:00Friends' birthday reminders (yearly repeat)
  • 17:30Existing personal-training reminders (adjustable)
  • 23:00Diary writing (private chat) + Notion sync
Note: times are based on Asia/Shanghai for a life-friendly rhythm.

Signal radar (HEARTBEAT)

  • GitHub Changelog RSS: Copilot / Agents / AI items marked [important]
  • Microsoft DevBlogs RSS: AI / Agent / Foundry / MCP items marked [important]
  • Weekly: client news + release matching (map new releases to client use cases)
  • Weekly: competitor AI updates (Google / Amazon / Alibaba / ByteDance / Tencent)

Why do reminders / signals need to pair with "write into the system"?

Reminder = trigger; write = memory. Only when both are coupled can a team review what it saw, what it did, and what should happen next.

Engineering details (the easiest place for teams to get burned)

These are not "showoff details"; they are what keep the system reliable over time.

Idempotency & deduplication

  • Query before Notion writes: if the same day / topic already exists, update instead of create
  • If duplicates happen, archive them instead of hard-deleting
  • External receipts must reflect the real write result

Permissions & auth

  • Azure Foundry image generation: RBAC dataActions (images/generations/action) is a common pitfall
  • Use Managed Identity (DefaultAzureCredential) first for transcription and image generation
  • Load environment variables through wrapper scripts to reduce random production breakage

Traceable artifacts (strongly recommended)

Raw transcript.txt Structured summary.json / blocks.json Media mood-YYYY-MM-DD.png Logs memory/YYYY-MM-DD.md
In team collaboration, "traceable" matters far more than "looks smart".

A minimum-viable rollout suggestion for teams

If you want to copy this setup, start with three things: get one workflow running end to end, get one reminder running, and make one write path idempotent.

Step 1: choose one high-frequency + high-value scenario

  • Client meeting audio -> transcript -> recap -> database
  • Or: weekly client news + release matching (great for sales / solutions)
Do not start with an "all-purpose assistant." Start with one system that can run stably for a long time.

Step 2: establish two hard guarantees (to preserve trust)

  • Write receipt guard: if the write fails, say exactly why
  • Artifact persistence: keep transcripts, summaries, images, and key JSON on disk
These two lines dramatically reduce the trust erosion of "AI said it did it, but it actually did not".

Step 3: then talk about lifestyle use cases

Lifestyle reminders (weather / almanac / birthdays / training) are wonderful, but they work best after the work system is already trusted — otherwise the result feels busy rather than dependable.
