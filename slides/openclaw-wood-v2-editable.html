生活搭子 + 工作操作系统

我不是把 AI 当成“回答问题的人”，而是把它养成一个会记忆、会执行、会主动提醒、会把结果写回系统的数字搭子。

这不是“我跟 AI 聊天”的故事，是“我把它养成系统”的故事

前半段全是调试、打补丁、对齐规则、重置 session、修 Notion、修 GitHub、修定时器；后半段才是稳定地享受成果。下面这几屏截图，基本就是我这段历程的原始现场。

从最早的长对话、调试、问答，到开始给它加规则、加技能、加执行 guard。

我的真实使用感受

  • 一开始它只是“会说”；后来我逼它变成“会做”。
  • 很多价值不是来自模型本身，而是来自规则、技能、定时、写回 Notion。
  • 最大的变化不是“更聪明”，而是“更靠谱”。
关键词 会话 关键词 记忆 关键词 技能 关键词 写回 关键词 定时
对我来说，OpenClaw 最有价值的地方不是“能回答”，而是“能在我的日常里占一个稳定位置”。

场景 ①：日记系统，真的会变成一种陪伴

我会让罗汉果每天写日记，而且不是模板流水账。它会结合当天真实聊天和经历，写得更像我的生活，再配一张“今日情绪图”，最后同步到 Notion。

日记管理入口

这个场景为什么重要

  • 它不只是记事，而是在帮我把情绪和生活留下来。
  • 图片让“那天的感觉”被保存下来，而不只是文字。
  • 写到 Notion 之后，就不是散落在聊天里的瞬时内容了。
这也是我调得最久的地方之一：它经常忘、经常跑偏、经常不够“像我”。但一旦调通，会很值。
关键词：真实聊天上下文、情绪图一致性、Notion 幂等更新、长期记忆。

场景 ②：客户会议录音 → Azure 语音服务 → 纪要 → Notion

这个是我工作里最实用的场景之一。把录音丢给它，它帮我走完转写、摘要、结构化纪要，再放回到客户管理的 Notion 笔记里。

IMG_0379：会议全文转写
IMG_0381：语音转文字能力展示

我最喜欢这条链路的点

  • 手机上把录音发过去就行，适合碎片化场景。
  • 会后纪要不再卡在“回头再整理”，因为它直接给出一页纸摘要。
  • 关键不是总结文本本身，而是写回 Notion，便于后续检索、复盘、追踪客户进展。
输入 音频 / 录音 处理 Azure Speech 输出 纪要 + Notion
一旦打通这一条，你就会开始习惯把“会后整理”外包给系统，而不是继续靠意志力。

场景 ③：它会主动推送东西给我，不只是等我来问

这块是我很喜欢的“生活/工作融合区”：技术博客、新闻、YouTube 周总结、客户与行业动态，都是系统按节奏推过来的。

它推给我的信息类型

  • 每天的技术博客和编程助手更新
  • 后面再叠加新闻、客户和行业进展

这类场景的关键，不是信息量多，而是要按主题分开，不然一个入口里什么都有会很乱。

场景 ④：生活管理也可以很顺手——天气、黄历、生日提醒

我会把一些琐碎但高频的小事交给它：早上的天气和“今天适合做什么/不适合做什么”，以及朋友生日提醒。

这些提醒很小，但很值

先让它把这些小事做好，你才会放心把更重要的事交给它。

场景 ⑤：单独开一个群，让它做我的健身教练

这个是我特别喜欢的用法：不是让它在主入口里兼任所有角色，而是单独给它一个身份、一个群聊、一个节奏。

IMG_0393：和健身教练一起设定训练计划
IMG_0398：每日计划完成后导入 Notion 记录

它在这个群里扮演的角色

  • 每周帮我订计划
  • 每天给训练单和提醒
  • 动作不懂时随时问
  • 训练反馈再写回 Notion，形成长期记录
如果所有事情都塞进同一个入口，最后只会越来越混乱。分角色、分群聊，是我目前觉得最有效的方式之一。

除了上面那些“主战场”，我还会顺手让它做很多碎但有用的事：高德路线、美食推荐、行程规划、东方财富查询、以及我懒得开电脑时帮我快速搭网页、部署网页。

为什么它后来越来越好用？因为我开始把它当“工程项目”养

真正让体验变好的，不是我突然找到某个神 prompt，而是我开始持续修：记忆、会话、技能、定时器、Notion、GitHub、权限、回执规则……

底层上我做了什么

  • 给它加长期记忆与 soul 规则
  • 修复 Notion / GitHub / Tavily / Azure 等鉴权
  • 把“重置会话”“记住这件事”做成口令
  • 把外部写入改成：必须真实执行才能回执

这背后的变化

需求是有限的，技术不一定有限。先抓住那个很小、但你真的想解决的问题。然后在调它的过程中，你会被迫一路学会更多东西。
从 对话 到 系统 从 热闹 到 可靠

最后，我最想分享给团队的 4 个感受

这不是“最佳实践大全”，而是我作为真实用户一路踩坑之后，最想留下来的四句话。

  1. 需求有限，而不是技术有限。 先从一个很小但真实想解决的问题开始，AI 会逼你一路去探索新的边界。
  2. 要接受失败和混乱。 比如日记这一块，我就调了很久，它会忘、会偏、会不按规范来。
  3. 要充分用好公司的 GitHub Copilot 资源。 模型新、能力全，这其实是很好的试验田。
  4. 不要所有事情都走同一个入口。 同一个上下文里塞太多话题很容易乱，最好按角色拆成不同代理 / 不同群聊。
我越来越相信：好的 AI 使用方式，不是“问得更花”，而是“把一个个小需求养成稳定的系统”。
